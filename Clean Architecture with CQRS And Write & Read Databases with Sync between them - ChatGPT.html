Yes - your approach is correct if Product.Id is generated by the database, for example SQL Server IDENTITY.

You need this order:

  1. Start transaction.
  2. Add product.
  3. SaveChangesAsync() to force EF Core to insert the product and get the generated Id.
  4. Build outbox message using product.Id.
  5. Add outbox message.
  6. SaveChangesAsync() again.
  7. Commit transaction.

The important point: the first SaveChangesAsync() does not mean final commit. As long as you are inside the same transaction, both the product insert and outbox insert will be rolled back if anything fails before CommitAsync().

Example:

await using var transaction =
    await _writeContext.Database.BeginTransactionAsync(cancellationToken);

try
{
    var product = new Product
    {
        Name = request.Name,
        Stock = request.Stock,
        UnitPrice = request.UnitPrice,
        ProductStatus = ProductStatuses.Draft
    };

    _writeContext.Products.Add(product);

    // Needed if Product.Id is database-generated
    await _writeContext.SaveChangesAsync(cancellationToken);

    var productCreatedEvent = new ProductCreatedEvent
    {
        Id = product.Id,
        Name = product.Name,
        Stock = product.Stock,
        UnitPrice = product.UnitPrice,
        ProductStatus = product.ProductStatus
    };

    var outboxMessage =
        OutboxMessage.BuildOutboxMessage(
            BrokerEventsEnum.ProductCreatedEvent,
            productCreatedEvent
        );

    _writeContext.OutboxMessages.Add(outboxMessage);

    await _writeContext.SaveChangesAsync(cancellationToken);

    await transaction.CommitAsync(cancellationToken);

    return product.Id;
}
catch
{
    await transaction.RollbackAsync(cancellationToken);
    throw;
}

So this is safe:

await _writeContext.SaveChangesAsync(cancellationToken);

before creating the outbox message, because the transaction has not committed yet.

The final result is atomic:

Product saved + Outbox saved   => commit
Product saved + Outbox failed  => rollback product too

But do not publish to RabbitMQ inside this transaction. The request handler should only save:

Product row
OutboxMessage row

Then a background service reads pending outbox messages and publishes them to RabbitMQ later.

If your Product.Id is a Guid, you can avoid the first SaveChangesAsync() by generating the ID yourself:

var product = new Product
{
    Id = Guid.NewGuid(),
    Name = request.Name,
    Stock = request.Stock,
    UnitPrice = request.UnitPrice,
    ProductStatus = ProductStatuses.Draft
};

var productCreatedEvent = new ProductCreatedEvent
{
    Id = product.Id,
    Name = product.Name,
    Stock = product.Stock,
    UnitPrice = product.UnitPrice,
    ProductStatus = product.ProductStatus
};

var outboxMessage =
    OutboxMessage.BuildOutboxMessage(
        BrokerEventsEnum.ProductCreatedEvent,
        productCreatedEvent
    );

_writeContext.Products.Add(product);
_writeContext.OutboxMessages.Add(outboxMessage);

await _writeContext.SaveChangesAsync(cancellationToken);

But if your ID is int identity, your current two-SaveChangesAsync() approach inside one transaction is the right one.

I’ll show the clean design: one singleton RabbitMQ connection for the whole app, but separate channels for producer and consumer because channels should not be shared concurrently.

Yes. The correct design is:

One shared RabbitMQ connection
    ├── producer channel
    └── consumer channel

Do not share the same channel between producer and consumer. RabbitMQ connections are meant to be long-lived; opening a connection per operation is inefficient and discouraged. Channels are lightweight logical connections over the same TCP connection, but a channel should not be used concurrently by multiple threads, especially for publishing. (RabbitMQ) (RabbitMQ) (RabbitMQ)

1. Create shared RabbitMQ connection provider

using Microsoft.Extensions.Options;
using RabbitMQ.Client;

namespace CQRS.Infrastructure.Brokers;

public interface IRabbitMqConnectionProvider
{
    Task<IConnection> GetConnectionAsync(CancellationToken cancellationToken = default);
    Task<IChannel> CreateChannelAsync(CancellationToken cancellationToken = default);
}

using Microsoft.Extensions.Options;
using RabbitMQ.Client;

namespace CQRS.Infrastructure.Brokers;

public sealed class RabbitMqConnectionProvider : IRabbitMqConnectionProvider, IAsyncDisposable
{
    private readonly RabbitMqOptions _options;
    private readonly SemaphoreSlim _lock = new(1, 1);

    private IConnection? _connection;

    public RabbitMqConnectionProvider(IOptions<RabbitMqOptions> options)
    {
        _options = options.Value;
    }

    public async Task<IConnection> GetConnectionAsync(CancellationToken cancellationToken = default)
    {
        if (_connection is { IsOpen: true })
            return _connection;

        await _lock.WaitAsync(cancellationToken);

        try
        {
            if (_connection is { IsOpen: true })
                return _connection;

            if (_connection is not null)
            {
                await _connection.DisposeAsync();
                _connection = null;
            }

            var factory = new ConnectionFactory
            {
                HostName = _options.HostName,
                Port = _options.Port,
                UserName = _options.UserName,
                Password = _options.Password,
                VirtualHost = _options.VirtualHost,

                AutomaticRecoveryEnabled = true,
                NetworkRecoveryInterval = TimeSpan.FromSeconds(10),

                ClientProvidedName = "CQRS.API"
            };

            _connection = await factory.CreateConnectionAsync(cancellationToken);

            return _connection;
        }
        finally
        {
            _lock.Release();
        }
    }

    public async Task<IChannel> CreateChannelAsync(CancellationToken cancellationToken = default)
    {
        var connection = await GetConnectionAsync(cancellationToken);

        return await connection.CreateChannelAsync(
            cancellationToken: cancellationToken
        );
    }

    public async ValueTask DisposeAsync()
    {
        if (_connection is not null)
        {
            await _connection.DisposeAsync();
        }

        _lock.Dispose();
    }
}

4. Producer uses its own channel

Example:

using System.Text;
using Microsoft.Extensions.Options;
using RabbitMQ.Client;

namespace CQRS.Infrastructure.Brokers;

public sealed class RabbitMqProducer : IAsyncDisposable
{
    private readonly IRabbitMqConnectionProvider _connectionProvider;
    private readonly RabbitMqOptions _options;

    private IChannel? _channel;
    private readonly SemaphoreSlim _publishLock = new(1, 1);

    public RabbitMqProducer(
        IRabbitMqConnectionProvider connectionProvider,
        IOptions<RabbitMqOptions> options)
    {
        _connectionProvider = connectionProvider;
        _options = options.Value;
    }

    public async Task PublishAsync(
        string routingKey,
        string message,
        CancellationToken cancellationToken = default)
    {
        _channel ??= await _connectionProvider.CreateChannelAsync(cancellationToken);

        await _publishLock.WaitAsync(cancellationToken);

        try
        {
            var body = Encoding.UTF8.GetBytes(message);

            await _channel.BasicPublishAsync(
                exchange: _options.ExchangeName,
                routingKey: routingKey,
                body: body,
                cancellationToken: cancellationToken);
        }
        finally
        {
            _publishLock.Release();
        }
    }

    public async ValueTask DisposeAsync()
    {
        if (_channel is not null)
            await _channel.DisposeAsync();

        _publishLock.Dispose();
    }
}

The _publishLock is important if this producer is used by multiple threads. RabbitMQ officially warns that concurrent publishing on the same channel can cause frame interleaving problems. (RabbitMQ)

Register producer:

builder.Services.AddSingleton<RabbitMqProducer>();

5. Consumer uses another dedicated channel

using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;
using Microsoft.Extensions.Options;
using RabbitMQ.Client;
using RabbitMQ.Client.Events;
using System.Text;

namespace CQRS.Infrastructure.Brokers;

public sealed class RabbitMqConsumerBackgroundService : BackgroundService
{
    private readonly IRabbitMqConnectionProvider _connectionProvider;
    private readonly RabbitMqOptions _options;
    private readonly ILogger<RabbitMqConsumerBackgroundService> _logger;

    private IChannel? _channel;

    public RabbitMqConsumerBackgroundService(
        IRabbitMqConnectionProvider connectionProvider,
        IOptions<RabbitMqOptions> options,
        ILogger<RabbitMqConsumerBackgroundService> logger)
    {
        _connectionProvider = connectionProvider;
        _options = options.Value;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _channel = await _connectionProvider.CreateChannelAsync(stoppingToken);

        await _channel.ExchangeDeclareAsync(
            exchange: _options.ExchangeName,
            type: ExchangeType.Topic,
            durable: true,
            autoDelete: false,
            cancellationToken: stoppingToken);

        await _channel.QueueDeclareAsync(
            queue: _options.QueueName,
            durable: true,
            exclusive: false,
            autoDelete: false,
            cancellationToken: stoppingToken);

        await _channel.QueueBindAsync(
            queue: _options.QueueName,
            exchange: _options.ExchangeName,
            routingKey: "product.*.v1",
            cancellationToken: stoppingToken);

        var consumer = new AsyncEventingBasicConsumer(_channel);

        consumer.ReceivedAsync += async (_, eventArgs) =>
        {
            try
            {
                var body = eventArgs.Body.ToArray();
                var message = Encoding.UTF8.GetString(body);

                _logger.LogInformation("Received RabbitMQ message: {Message}", message);

                // TODO: handle your event here

                await _channel.BasicAckAsync(
                    deliveryTag: eventArgs.DeliveryTag,
                    multiple: false,
                    cancellationToken: stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Error while consuming RabbitMQ message");

                await _channel.BasicNackAsync(
                    deliveryTag: eventArgs.DeliveryTag,
                    multiple: false,
                    requeue: true,
                    cancellationToken: stoppingToken);
            }
        };

        await _channel.BasicConsumeAsync(
            queue: _options.QueueName,
            autoAck: false,
            consumer: consumer,
            cancellationToken: stoppingToken);
    }

    public override async Task StopAsync(CancellationToken cancellationToken)
    {
        if (_channel is not null)
            await _channel.DisposeAsync();

        await base.StopAsync(cancellationToken);
    }
}

Register consumer:

builder.Services.AddHostedService<RabbitMqConsumerBackgroundService>();

Final structure

RabbitMqConnectionProvider Singleton
    |
    |-- RabbitMqProducer
    |       uses producer channel
    |
    |-- RabbitMqConsumerBackgroundService
            uses consumer channel

So the rule is:

Share IConnection
Do not share IChannel between producer and consumer

This gives you one TCP connection to RabbitMQ, but safe independent channels for publishing and consuming.

You’re right to question that. The need for _publishLock depends on the lifetime of the channel, not only the lifetime of the producer service.

Correct - in your current design, you may not need _publishLock.

The rule is:

_lock is needed only when the SAME IChannel instance can be used concurrently.

RabbitMQ says IChannel should not be used by more than one thread at the same time, and concurrent publishing on the same channel can cause frame interleaving. But that applies when the same channel object is reused by multiple threads. (RabbitMQ)

Case 1: You create a new channel every publish

Example:

public async Task PublishAsync(
    string routingKey,
    string message,
    CancellationToken cancellationToken)
{
    await using var channel =
        await _connectionProvider.CreateChannelAsync(cancellationToken);

    var body = Encoding.UTF8.GetBytes(message);

    await channel.BasicPublishAsync(
        exchange: _options.ExchangeName,
        routingKey: routingKey,
        body: body,
        cancellationToken: cancellationToken);
}

Here:

Each PublishAsync call gets its own channel.
No shared channel.
No concurrent usage of same channel.
No _publishLock needed.

So yes, in this design, _publishLock is unnecessary.

Case 2: Producer has a field-level channel

Example:

private IChannel? _channel;

public async Task PublishAsync(...)
{
    _channel ??= await _connectionProvider.CreateChannelAsync();

    await _channel.BasicPublishAsync(...);
}

Here _channel is stored inside the producer object.

Now the question becomes: can two threads call PublishAsync on the same producer instance at the same time?

If yes, then both could use the same _channel concurrently.

In that case, you need:

private readonly SemaphoreSlim _publishLock = new(1, 1);

So:

Shared channel field + possible concurrent PublishAsync calls = need lock.

About scoped producer

If your producer is registered as scoped:

builder.Services.AddScoped<IRabbitMqProducer, RabbitMqProducer>();

Then ASP.NET Core creates one producer instance per HTTP request.

Usually, one request runs its code sequentially, so this is often safe.

But technically, scoped does not guarantee no concurrency. For example, inside one request you could do:

await Task.WhenAll(
    _producer.PublishAsync(...),
    _producer.PublishAsync(...)
);

Then both calls use the same scoped producer instance. If that producer has a shared _channel field, you still need the lock.

Best simple answer

If you want the safest and simplest design, do this:

public async Task PublishAsync(
    string routingKey,
    string message,
    CancellationToken cancellationToken)
{
    await using var channel =
        await _connectionProvider.CreateChannelAsync(cancellationToken);

    var body = Encoding.UTF8.GetBytes(message);

    await channel.BasicPublishAsync(
        exchange: _options.ExchangeName,
        routingKey: routingKey,
        body: body,
        cancellationToken: cancellationToken);
}

Then register:

builder.Services.AddSingleton<IRabbitMqConnectionProvider, RabbitMqConnectionProvider>();
builder.Services.AddScoped<IRabbitMqProducer, RabbitMqProducer>();

With this design:

Connection = shared singleton
Channel = created per publish
_publishLock = not needed

But for better performance

Creating a channel is much cheaper than creating a TCP connection, because RabbitMQ channels are lightweight logical connections over a shared AMQP connection. (RabbitMQ, RabbitMQ)

Still, for high-volume publishing, creating a channel for every single message is not ideal.

For your current CQRS/outbox project, I would recommend this design:

RabbitMqConnectionProvider = Singleton
OutboxPublisherBackgroundService = Singleton hosted service
ConsumerBackgroundService = Singleton hosted service

Each background service creates its own long-lived channel.

Like this:

Shared RabbitMQ connection
    ├── Outbox publisher channel
    └── Consumer channel

If your outbox publisher processes messages one by one, no lock is needed because only one thread uses that publisher channel.

If your outbox publisher processes messages in parallel, then either:

Option A: one channel per worker
Option B: shared channel + SemaphoreSlim

Important correction

The RabbitMQ consumer itself should not be scoped.

This is better:

builder.Services.AddHostedService<RabbitMqConsumerBackgroundService>();

A BackgroundService is effectively singleton. If it needs scoped services like DbContext, create a scope inside the message handler:

using var scope = _serviceScopeFactory.CreateScope();

var handler = scope.ServiceProvider
    .GetRequiredService<IYourEventHandler>();

await handler.HandleAsync(message, cancellationToken);

So the clean final rule for your app is:

Connection provider: Singleton
Producer/outbox background service: Hosted service
Consumer background service: Hosted service
DbContext/event handlers: Scoped
Channel: one per producer/consumer worker
Lock: only if the same channel can be used concurrently
